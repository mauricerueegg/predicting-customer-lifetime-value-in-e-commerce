# 02 – Feature Engineering

## Why Feature Engineering Matters

The raw dataset contains behavioural metrics that individually measure different aspects of customer activity. However, the CLV generating formula combines these in ways that create meaningful composite signals. Feature engineering helps:

1. **Expose patterns** that are implicit in the raw data
2. **Improve linear model performance** by approximating interaction effects
3. **Provide domain-meaningful inputs** that are easier to interpret via SHAP

---

## Encoding Categorical Features

### One-Hot Encoding

Both categorical columns must be converted to numeric form before modelling.

**`customer_segment`** → 4 binary columns:
- `segment_new`, `segment_occasional`, `segment_loyal`, `segment_premium`
- The segment has **non-ordinal** categories — one-hot encoding is appropriate
- (No "premium > loyal > occasional > new" ordering should be assumed)

**`preferred_device`** → 3 binary columns:
- `device_mobile`, `device_desktop`, `device_tablet`
- Again, non-ordinal — one-hot encoding

**Implementation:**
```python
df_encoded = pd.get_dummies(df, columns=["customer_segment", "preferred_device"], drop_first=False)
```

**Note:** We keep all dummy columns (no drop_first) so SHAP values are interpretable per segment/device. Multicollinearity is acceptable for tree models; for Linear Regression, regularisation can handle it.

---

## Engineered Features

### 1. `total_spend_proxy`

```python
total_spend_proxy = purchase_count_last_12m * avg_order_value_chf
```

**Rationale:** The CLV formula combines `purchase_count_last_12m * 42` and `avg_order_value_chf * 5.2` independently. However, in reality total spending is a product of frequency × order size. This interaction feature helps linear models approximate the multiplicative relationship and gives tree models a useful split criterion.

**Domain meaning:** Estimated gross revenue generated by the customer in the last 12 months.

---

### 2. `engagement_score`

```python
engagement_score = website_visits_last_3m * email_click_rate
```

**Rationale:** Both `website_visits_last_3m` and `email_click_rate` contribute positively to CLV. A customer who visits frequently AND engages with emails is more valuable than one who does only one. The product captures this compound engagement signal.

**Domain meaning:** Combined digital engagement index — high only when both visit frequency and email responsiveness are elevated.

---

## Feature Scaling

Tree-based models (Random Forest, XGBoost) are **scale-invariant** — they split on feature thresholds, not magnitudes. No scaling is needed.

Linear Regression is **scale-sensitive** — features with large ranges (e.g., `total_spend_proxy` can be up to 18000) will dominate coefficients. We use `StandardScaler` (zero mean, unit variance) applied **only to the training set** to avoid data leakage.

```python
from sklearn.preprocessing import StandardScaler

scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)  # fit on train only
X_test_scaled  = scaler.transform(X_test)        # transform test with same scaler
```

## What We Do NOT Engineer

- **Log transforms of the target:** The CLV values are clipped at 20 CHF and right-skewed. A log transform could stabilise variance, but would make RMSE/MAE less interpretable in CHF units. We keep the original scale and note the skew in Section 2.
- **Polynomial features:** Tree-based models handle non-linearities natively. Adding polynomial terms would primarily benefit Linear Regression but risk overfitting and hurts interpretability.
- **Binning:** Discretising continuous features (e.g., tenure into buckets) loses information. Tree models find optimal splits automatically.
